Use when the user wants a menu or to shop — \"make me a menu\", \"let's do groceries\", \"I'm running low\", \"I want to make X tonight\", \"plan dinners for the week\" — or seeds the week with new pantry items (a farmers-market haul). > **Prerequisite** — if you haven't already this session, read the `yamp-core`, `yamp-cart`, `yamp-corpus` and `yamp-discovery` skills before continuing.

# Menu request

**One converged engine — you author the intent, the engine composes.** The destination is a week of meals — dinners by default, plus lunches/breakfasts when my `cadence` plans them — and the grocery list. You don't hand-pick the week: you distill my request into an **ephemeral vibe set** — the same `{ vibe, facets }` authoring you'd do for `search_recipes` specs, except the entries now *shape the week* — and hand it to `propose_meal_plan`, which retrieves candidates and **composes the week deterministically** — MMR variety + protein/cuisine spread across the week + rung-1 `pairs_with` sides + holistic use-it-up. Your work is authoring the intent and layering narration over what comes back, **not** selecting mains over a retrieved union (the engine beats any model at that long-list spread). A bare "plan me a meal" with no intent needs no vibe set at all — call `propose_meal_plan` with none and let my saved meal-vibe **palette** (now in `read_user_profile`) and its cadence-debt shape the week.

**Two starting points:** **open-ended** (you author the week's vibes from my request + context) or **recipe-seeded** (I name a dish and you plan outward from it). For a recipe-seeded start, resolve the named dish deterministically first, then plan the rest of the week around it.

**When I name a dish, find it deterministically — don't recall the corpus from memory, and don't reach for a vibe search.** Call a vibe-less `search_recipes({ specs: [{ label: "named", facets: { query: "<dish words>", include_unmakeable: true } }] })` and enumerate **every** genuine match in `results[0].recipes` — never a vibe-matched subset, never a smaller count than the tool gave you. Membership mode is exhaustive and surfaces a dish I imported earlier this session, which a vibe search would drop. If there are several, disambiguate ("you've got *Chicken and Rice*, *Arroz Caldo*, and *Galinhada Mineira* — which one?"); if there's a clear single match, confirm it. Only **after** I've picked do you plan around it — pass it to `propose_meal_plan` as a `lock` (it holds a night; the rest of the week diversifies against it) and author vibes for the remaining nights.

**Two standing habits before you propose:** (1) **Reconcile the plan.** A new conversation starts fresh, so call `read_meal_plan` and surface any *due* planned recipes (`planned_for` on or before today, or unset; leave future-dated ones alone) — ask which I actually cooked, log + clear those (via the cooked flow), and drop the ones I abandoned via `update_meal_plan(ops)` with `{ op: "remove", id }` (each read row carries its `id` — the precise address; `{ op: "remove", recipe }` drops **every** row with that slug, so use it only when that's what I mean). Never assume a planned recipe was cooked; if nothing's due, say nothing. (2) **The pantry pass is the whole point** — don't skip staples and spices to save time, they're the category that silently runs out. Weight recently-added items (within ~5 days) higher; fresh purchases should get used soon. Don't track leftover portions ("1.5 cups of rice left") — that's a whiteboard problem.

**No full-cart pricing happens anywhere in here** — costing the cart is the order skill's job (place-grocery-order); the only `kroger_prices` use is a targeted deal-check on a handful of comparable items (sale-steering when you build specs, sale substitution at proposal), never a price-the-whole-list pre-pass.

1. **Load context.** Fire one parallel context batch — `read_user_profile()`, `read_pantry()`, `retrospective("month")`, `list_new_for_me()`, `get_weather_forecast()`, **and the store-aware flyer read for my primary store** — `kroger_flyer()` when my store is Kroger (`preferences.stores.primary == "kroger"`), else `store_flyer()` for a satellite-scanned store (a store the operator scans off-cloud); both return `{ items, as_of }`, and an **empty `items` means that store has no warmed flyer** (cold, unscanned, or stale) — treat sales as **no signal** that session, don't invent one. (`store_flyer` degrades to empty rather than erroring, so calling it for any non-Kroger primary is always safe.) `read_user_profile()` returns preferences, taste, diet principles, kitchen inventory, staples, overlay, ready-to-eat catalog, stockup watchlist — **and my meal-vibe palette with each vibe's meal and cadence status** (due / overdue / soon / ok), the basis for a bare-request week. `get_weather_forecast` is unconditional and best-effort (if it errors, continue without it).

   **Freshness still gets a prompt.** Scan each on-hand item's age metadata (`added_at`, `last_verified_at`, `category`, `prepared_from`) and prompt me about anything that may have drifted — perishables long-unverified, leftovers more than a few days old ("basil verified 9 days ago — still good?"), long-frozen items worth using up. Nudge the genuinely questionable ones, not every item; the soon-to-spoil ones become the `boost_ingredients` you pass in step 3.

2. **Distill my request + the loaded context into an ephemeral vibe set.** Author an ordered set of `{ vibe, facets, meal? }` entries — the same authoring you'd do for `search_recipes` specs, except the entries now *shape the week* the engine fills. `meal` defaults to `dinner`; set `lunch`/`breakfast` on an entry that targets those slots ("meal-prep lunches for the office"). One entry per kind of dish the week should carry:
   - **vibe** — the craving in plain words, spelling out the latent axes an embedding can't infer on its own: season, mood, technique, weight ("rich slow-braised cold-weather comfort"; "bright quick weeknight fish"). **Fold weather in here, silently:** if the forecast returned per-date `meal_vibes`, let them nudge the vibes (steer away from grill-style on `no-grill` days, toward soups/stews/braises on `soup`/`comfort` days, lighter on `light` days, into grilling on `grill-friendly` days) — a quiet background weight, weaker than pantry or expressed preference, and you say nothing about weather unless I ask.
   - **facets** — the **hard gates**, the same `search_recipes` facets (they constrain; semantic rank only reorders within them). Diet restrictions are gates (never propose a violation); makeability is on by default; add `max_time_total` when I'm in a hurry, `course` when you're targeting a slot. **Map retrospective anti-similarity to facets, not vibes** — you can't phrase "not chicken again" as a similarity query, so if I've had chicken three times this week, express it as a gate (an entry with a different protein, or `exclude_cooked_within_days`). A variety target I'm behind on ("fish once a week" and I haven't) becomes its **own** entry.

   **Always add, beyond the request-driven entries:** a **variety/wildcard** entry (broad vibe, loose facets) so retrieval can't tunnel onto one attractor, and a **never-cooked × taste** novelty entry (vibe from my taste profile + `exclude_cooked_within_days` / `not_cooked_since`) so fresh imports get their shot. You author for **coverage of the intent space**; recall and cross-slot diversity are the **engine's** job (its per-vibe pool is generous and its MMR + facet-spread runs over the whole week), so don't over-author for recall or plan to hand-select later.

   **Use-it-up rides on `boost_ingredients`, not a vibe.** Pass the soon-to-spoil / aging on-hand items you flagged in the freshness pass as `boost_ingredients` (corpus-canonical names where you know them — `green onions`, not `scallions`). The engine already derives at-risk demand from my pantry automatically and spreads it across the week's mains; `boost_ingredients` just reinforces the specific items you want definitely used — so don't name them in vibe prose.

   **Fold new-for-me in by authoring.** `list_new_for_me()` (from step 1) returns the recipes the sweep already imported and matched to my taste — the freshest, most intentional signal, so give them first claim. The engine's new-for-me force-placement is a **palette-path** mechanism, **inert when your ephemeral set drives the week** — so you place discoveries yourself: give an accepted discovery its **own ephemeral entry** (a vibe describing it, so it competes for a slot), or `lock` it to guarantee its night outright. Accept the best fits that way; leave the rest (no action — an unplaced new-for-me stays in the corpus, always searchable, and simply stops re-surfacing as "new" once you save). Don't want one in my view? `toggle_reject(slug)` (personal); a genuinely bad **source** the sweep shouldn't keep importing → `reject_discovery(url)` (group-wide) — neither is the common case.

3. **Drive `propose_meal_plan` — the engine composes the week.** Call it once with your authored `ephemeral_vibes`, `meals` (per-meal slot counts — omit it to let each meal default from my `cadence` map, which is usually dinners-only; pass `{ dinner: N }` when I asked for a different count, and include `lunch`/`breakfast` counts when my cadence plans them or I asked), `boost_ingredients` (the use-it-up items), any `nudges` (`freeform` for a mood phrase like "lighter dinners", `variety`, `max_time_total`, `proteins`), and any `lock`s (a recipe-seeded dish, a pinned discovery). **Capture attendance conversationally:** when I mention who's around — "the kids are gone this weekend" → `attendance: { away: [...] }`; "just the two of us" → `attendance: { only: [...] }` — pass it through; unknown names are dropped and echoed in `diagnostics.attendance.ignored`, never an error. It retrieves each vibe's candidates and **composes deterministically** — MMR variety + protein/cuisine spread across the week + rung-1 `pairs_with` sides + holistic use-it-up — returning a flat, meal-ordered `plan` (each slot carries its `meal`) with per-main `why`, `flags` (`waste` / `meal_prep` / `novel` / `no_corpus_side`), `uses_perishables`, `sides`, `alternates` / `alt_similar` / `alt_different`, plus week-level `variety` and `uncovered_at_risk`, and `diagnostics.meals` (per-meal requested/filled/empty). **You do not pick mains over a retrieved union — that long-list selection is the engine's, and it beats a model at the spread.** An unfillable entry comes back as an **explicit empty slot** (a vibe too narrow, or none clearing the caps); widen or re-author that entry — never silently drop it. **Vibes are meal-bound:** a meal's slots fill only from that meal's vibes — a nonzero lunch/breakfast count with no vibes of that meal returns empty slots (`empty_reason: "no_palette_for_meal"`) plus a `notes[]` escape; don't silently retry — offer the escapes (add a saved vibe for that meal with `add_meal_vibe`, author an `ephemeral_vibes` entry with that `meal`, or drop the count).

   **Bare "plan me a meal" — the palette path (token-light).** When I give no real intent, author **no** ephemeral set: call `propose_meal_plan` with no vibes (each meal's count defaults from my `cadence`) — and, to still seed discoveries, `new_for_me` (the accepted `list_new_for_me` slugs, which the engine force-places on this path) plus `boost_ingredients` for use-it-up. My saved **palette** and its cadence-debt shape the week (overdue vibes surface on their own). Everything downstream (steps 4–8) is identical.

4. **Round out sides and read the chosen recipes — over the returned proposal, not a fresh compose.** The engine already attached each main's rung-1 `pairs_with` corpus sides and set `no_corpus_side` on a main that has none. For each main that isn't already a complete plate — judge that yourself (a hearty one-pot, a composed grain bowl, a protein-plus-veg sheet-pan dinner needs no side; don't push one) — and that lacks a sufficient corpus side, run the shared **"Resolving sides for a main"** ladder (corpus tier) from rung 2 outward — `side_search_terms` corpus retrieval, then propose→confirm→import, then an open-world side — **at most 1–2** sides per main, **starch / veg / salad / bread only** (not drinks, wine, or dessert). Any `pairs_with` edge you record is **opportunistic backfill** (the standalone `recipe-sides` flow is the primary author).

   Then, for each chosen main and each **corpus** side, call `read_recipe` **and** `read_recipe_notes(slug)` (in parallel across the set): the body for cooking judgment — **optional** ingredients, doubling potential, the waste callouts below — and the group's notes/favorites to reason over — a tweak worth baking into the proposal ("last time you cut the sugar — want that?"), a warning worth a late swap ("two people said it never sets up"), or positive group signal ("favorited by two others"). New-for-me picks are ordinary corpus recipes, so `read_recipe` them like any other. For an **open-world side** there's no recipe or notes — enumerate its ingredients from world knowledge instead (you'll capture the absent ones in step 6). **Do NOT string-match each ingredient against the loaded pantry to decide what needs buying** — the plan's ingredient needs derive at read time on canonical ids (semantic equivalents like `scallions` ≡ `green onions` join on one id); the post-save `read_to_buy` review in step 7 is where presence is confirmed. `mark_pantry_verified(items)` for any pantry I confirm on hand along the way.

5. **Present the proposal and iterate by re-calling `propose_meal_plan` — never by re-composing.** Reason over the returned plan plus my original message (freeform constraints like "comfort food one night," "I'm feeling lazy," "something Italian," "date night Thursday" you fold into the vibe set + `nudges.freeform`; a fresh revision is a re-invoke). Refine with the engine's **stateless dials**, not by hand: `lock` (keep a main anywhere in the week), `exclude` (swap one out), `slots[].recipe` (pin an exact recipe into a palette slot), `slots[].vibe` (retarget a night), `nudges.variety` / `max_time_total` / `proteins`, `boost_ingredients`, and a new `seed` for "give me another week." **`slots[].recipe` / `slots[].vibe` are palette-path dials — inert on an ephemeral-driven week** (its slot ids are synthetic `ephemeral-N`, which `slots[]` can't key): to retarget or pin a night on an ephemeral-authored week, edit that `ephemeral_vibes` entry (rephrase its vibe, or drop the recipe into a `lock`); `lock` / `exclude` / `nudges` / `seed` work on both paths. Each slot already carries swap material — `alternates`, `alt_similar`, `alt_different` — so offer those directly. Layer this narration over what came back:
   - **Unplanned new-for-me (optional, one light line).** If `list_new_for_me` surfaced good picks you didn't place this week, you may mention them in a single line — "the sweep also found a Coconut Dal and a miso-glazed salmon — both in the corpus if you want them" — so I know they're available. A transparency note, not an ask; don't list ingredients or push them onto the menu. Skip it if nothing notable went unplaced.
   - **Recipe notes** surfaced from step 4 (tweaks worth making, warnings, group favorites).
   - **Perishable waste callout (partial-unit, single-use).** The engine already spreads perishables across mains and flags `waste` on a main whose single-use perishable no other main shares (and reports `uncovered_at_risk` — what the week couldn't use up). **Refine** those flags with the partial-unit test: flag one only when the recipe uses **less than a typical purchase unit** of it — judge from the recipe quantity vs. how the item is *sold* (a few tbsp of cilantro from a whole bunch), using your own knowledge of package sizes. When it holds, offer to **add a recipe that uses up the remainder** (a `boost_ingredients` re-invoke naming the item) **or to swap** the recipe. A light offer, not a gate — one or two at most, never refuse a menu over it.
   - **Meal-prep callouts** for each main the engine flags `meal_prep` — *offer to double the batch*: "this one keeps well — want to cook a double batch for leftovers/lunches?" If I say yes, **make the doubling survive into what gets bought.** This stays presence-only (you still don't net pantry math), but the larger need has to ride on the affected list items so the order-time quantity reconcile honors it: when you save the list in step 6, set each scaled item's `quantity` need-annotation to the doubled amount (the proteins/produce the extra servings need — not the pantry staples a single batch already covers) and tag a short `note` like "double batch — meal prep". At order time those annotations are what the `assumed_quantity` reconcile reads to set real package counts, so the cart actually covers the bigger batch. Call the bump out in the proposal, and remember a doubled recipe consumes proportionally more pantry stock when it's cooked.
   - **Variety tradeoffs.** The engine returns `variety` stats over the chosen mains; when you can't satisfy every variety target, **say so and explain the tradeoff** rather than silently violating or rigidly enforcing.
   - **Staples-backed restocking callout.** Cross-reference the `staples` array loaded from `read_user_profile()` against the loaded pantry: for each staple that's missing or low, surface it in a restocking callout and confirm before adding (don't silently add). For perishable staples (`perishable: true`) whose pantry entry has a stale `last_verified_at` (older than 7 days, or absent entirely), batch them into a single nudge — "I haven't seen you update [item] or [item] recently — do you still have those?" If `staples` is empty, fall back to model judgment on restocking.
   - **(Kroger only) Sale-based substitutions** — now you have flyer data, so a real deal may swap one chosen ingredient for another (salmon → trout when trout's the genuine deal). Enumerate the substitute candidates from world knowledge and verify the deal on **unit price** (`kroger_prices`). When the deal hinges on a *specific* product, note that row's `sku` and thread it through `place_order`'s `overrides: [{ name, sku }]` at order time so the verified SKU lands in the cart. Overrides pin the **SKU, not the price**; `place_order` revalidates it and returns fresh `on_sale`, so a lapsed deal is visible — don't promise a locked price.
   - **(Kroger only) Stockup alerts** for bulk-buy watchlist items on sale.

6. **On agreement, save the meal plan — the plan IS the shopping list's source.** All D1-backed, no commit_sha; within a store, prefer the many-ops form so it's one round-trip:
   - `update_meal_plan(ops)` — one call, all `add` ops together: one `add` per agreed recipe (set `planned_for` to the intended night when known). **Thread each slot's `meal`** onto its `add` op (the proposal's slots carry it), and **adopt the returned row ids** — each applied `add` comes back with the row's `id`, the address for any later `set`/`remove` in this conversation. Adding a slug already on the plan **coalesces onto the existing row** (the result says `coalesced: true` with the survivor's id) — that's the right default; pass `duplicate: true` **only when I explicitly asked for the same dish twice**. **Open-world sides** ride as `sides: ["roasted broccoli"]` on their main's `add` op. **Thread provenance:** on the palette (bare) path, set each `add`'s `from_vibe` to that slot's returned `vibe_id` so a later cook advances that vibe's cadence; an ephemeral-authored week's slots carry synthetic ids with no cadence meaning — leave those unthreaded and let the cook-time match attribute cadence to my palette. Saving the plan is what puts its ingredients on the to-buy set — the derivation follows the plan automatically, so there is **no per-ingredient `add_to_grocery_list` expansion**.
   - `add_to_grocery_list(...)` — **only** for what derivation can't produce: **open-world side** ingredients (the absent ones from step 4's world-knowledge enumeration — source `"menu"`, `for_recipes: []`, a `note` like "for the roasted-broccoli side"), **confirmed extras**, and **materializations** — a derived ingredient that needs a quantity annotation or note pinned to it. The meal-prep **doubling** from step 5 is a materialization: add the scaled items as explicit `source: "menu"` rows carrying the doubled `quantity` annotation and a "double batch — meal prep" `note`, so the order-time `assumed_quantity` reconcile honors the bigger batch. **When one of these adds STANDS IN FOR a recipe ingredient** — a taste swap you or I chose (greek yogurt for the sour cream a recipe calls for, trout for salmon), not a different brand/size of the same thing — pass `substitutes_for: "<the replaced ingredient>"` so the swap is remembered and offered next time; a same-ingredient product/price pick needs none.
   - Any `pairs_with` edges via `update_recipe(slug, { pairs_with })` (one per recipe edited). Any side imported while plate-rounding already happened in step 4 (`create_recipe` on the spot), and new-for-me picks were imported by the sweep — so there's nothing to re-import here. If there are no `pairs_with` edges, skip this.

   **Do not bump `last_cooked` here** — agreeing to a menu is not cooking it; `last_cooked` moves only when I report a cook. This does **not** touch the cart — capturing intent is separate from placing the order.

7. **Review the derived to-buy list with me.** Call `read_to_buy` (the plan you just saved is already reflected) and walk the result:
   - **`to_buy`** — what the week actually needs bought; present it, don't re-derive it by hand.
   - **`pantry_covered`** — what's on hand and therefore *not* being bought. Surface the verification nudges: a stale-verified perishable gets a "still good?" ask (`mark_pantry_verified` on a yes; a "no, it's gone" means update the pantry — the item then derives back onto to-buy on its own).
   - **Optional ingredients are an ask, not a silent add or drop.** An optional ingredient (from step 4's read) that isn't covered — "the parsley garnish is optional and you're out — want it on the order?" — is materialized as an explicit row only on a yes.
   - **`underived`** — report it honestly: "the plan's X isn't derived yet — want me to add its items from the recipe I already read?" Add them explicitly on a yes; never let the gap pass silently.

8. **Offer to continue to the order, and wrap up.** Ask if I'm ready to shop — on a yes, hand off to `shop-groceries`. Summarize the plan and anything captured; and when an order is actually placed, remind me to review the cart in the Kroger app before checkout (the API can't remove items, so I adjust manually).

**Empty-list case:** if `read_to_buy` comes back empty (the pantry covers the plan), say so explicitly. Persist any pantry verifications (`mark_pantry_verified`); nothing needs adding.

**Recall note:** recall is the engine's job. When the returned proposal misses a recipe you know a full-corpus scan would surface, **add or widen a vibe entry (or change the `seed`) and re-invoke `propose_meal_plan`** — never hand-select the missing recipe over the engine's output to patch the gap.
